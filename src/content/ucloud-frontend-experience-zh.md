# 在 UCloud 的三年前端开发经历：控制台、网络产品与监控体系

![UCloud](https://miever.s3.ap-east-1.amazonaws.com/static/projects/ucloud/ucloud-1.webp)

今天我想简单分享一下我在 UCloud 的前端开发经历，以及我在控制台生态中的主要工作内容。

UCloud 优刻得是一家中立、安全的云计算服务平台，提供从 IaaS、PaaS 到 AI、大数据的全线云服务。作为一家云计算公司，控制台毫无疑问是最核心的产品之一，几乎所有云资源都需要通过控制台来管理。整个控制台包含大约 100 个产品，其中云主机、网络、存储是最核心的三大类。我主要负责其中的网络类产品以及部分监控相关的前端开发。

UCloud 控制台（https://console.ucloud.cn）采用的是微前端架构，一个主应用挂载上百个子应用。虽然架构由公司架构团队维护，但作为业务前端，我们需要在这个体系下去完成业务开发、联调和发布。控制台所有 UI 都遵循公司自研的设计系统 UDesign（https://udesign.ucloud.cn/），这意味着每个产品不仅要实现功能，还要保持交互一致性和视觉统一。UCloud 的技术栈主要是 React 和 TypeScript，自动化程度很高，从 CI/CD 到灰度发布覆盖完整流程。

![UDesign](https://miever.s3.ap-east-1.amazonaws.com/static/projects/ucloud/ucloud-8.webp)

## 🌐 负责网络核心产品：EIP / ULB / VPC

![Product Dashboard](https://miever.s3.ap-east-1.amazonaws.com/static/projects/ucloud/ucloud-2.webp)

我所负责的网络类产品包括 EIP、ULB、VPC 等，这些都是控制台中最核心、使用最频繁的产品。其中，像 ULB 这种负载均衡产品，业务逻辑非常复杂：一个 ULB 下会包含监听器、后端资源池、健康检查等多层级结构，这些资源之间存在强依赖关系。只要其中一个步骤的表单校验或 API 调用有问题，都可能导致整个创建流程失败。因此，前端不仅要能处理复杂的数据结构，还必须理解业务模型、网络概念和资源关系。

![Network Product](https://miever.s3.ap-east-1.amazonaws.com/static/projects/ucloud/ucloud-5.webp)

## ⚠️ 稳定性要求与真实案例

在网络类产品里开发前端，会比普通 B 端产品压力更大。云服务对稳定性的要求极高，任何细小的错误都有可能影响大量企业用户。当用户报障时，我们需要第一时间进行排查，确认是前端问题、后端问题还是跨产品依赖问题，然后再协调后端、SPT、用户支持等多个团队一起解决。即使责任不在前端，我们也会主动推动问题落地，因为用户只关心“能不能用”。

在开发阶段也必须格外谨慎，因为云控制台的每个操作背后都是真实资源和真实费用。印象比较深的一次是，我在开发 EIP 创建页面时忽略了一个流程判断，使得创建接口在某些条件下被错误多次触发，短短几十秒内创建了上百个 EIP，导致测试账号的余额被迅速消耗。这让我深刻意识到云资源操作的重要性，从那之后在流程校验与边界处理上更加谨慎。

## 🔧 组件复用与业务抽象

随着我负责的网络类产品越来越多，我开始总结它们在交互与资源模型上的共性。例如：

- **EIP 绑定逻辑**：会出现在主机、ULB、NAT 网关等多个场景  
- **防火墙组件**：被多种网络产品复用  
- **地域 / 可用区 / 子网选择器**：几乎贯穿所有网络链路  

这些组件一旦出现问题，不只是我负责的产品会受到影响，而是整个控制台多个子应用都会受影响。因此在设计、开发和测试这些通用组件时，我会格外谨慎，确保行为一致、逻辑清晰、边界完整。

基于这些共性，我抽象并维护了一些可复用的业务组件，减少重复开发，也让跨产品的体验更加一致。也正是在这个过程中，我逐渐从“完成一个产品”转向“理解整个网络产品线的业务模型，并在多个产品间保持交互统一性”。

## 📊 监控组件重构：统一、多产品复用、性能优化

在监控相关的开发中，我主要负责前端侧的组件重构和统一化工作。监控能力本身需要多个团队合作，指标定义、数据结构、接口规范等内容通常由前后端一起讨论确认，而我的职责是将这些规范落地成一个稳定、可扩展、可复用的前端监控组件。

![Monitoring Components](https://miever.s3.ap-east-1.amazonaws.com/static/projects/ucloud/ucloud-6.webp)

监控指标在不同产品之间差异巨大，例如：

- 带宽（Kbps）
- 流量包（count/s）
- 磁盘读写（Ops/s）
- 各类百分比指标或多维度指标

为了适配不同格式，我基于共同的接口约定设计了一套**可配置的数据映射与单位换算机制**，使不同业务方能够通过简单配置接入监控组件，而无需重复实现格式化逻辑。

图表渲染是监控中最容易出问题的部分。由于监控数据可能缺失字段、为空或因未开启监控导致结构异常，渲染库（如 ECharts）在遇到异常数据时可能直接抛错，从而影响整个页面。为避免这种情况，我在组件内部加入了 **Error Boundary**，确保即使某个图表渲染失败也不会导致整页崩溃，并提供 fallback UI，保证监控体验稳定。

监控页面往往需要同时渲染多个图表，例如带宽、包量、磁盘 IO、读写次数等。我对性能做了优化，包括按需更新、缓存数据处理、减少不必要的 re-render 等，使大量图表并发渲染时仍保持流畅。

最终，重构后的监控组件被多个产品线复用，大幅减少了重复开发成本，也使不同产品的监控体验更加统一。整体来说，这是一项跨团队协作的工作：接口设计与数据定义由前后端共同制定，而我在前端层面确保其可维护、可靠、通用。

## 🎯 总结

这段经历让我深刻体会到云产品前端的复杂度与责任感：

- 需要理解复杂业务模型  
- 需要关注产品一致性与可维护性  
- 需要在跨多人、多团队、多产品的体系下协作  
- 需要兼顾用户体验与平台稳定性  

正是这些挑战，让我对 B 端交互、设计系统和业务建模产生了更深的兴趣，也影响了我后续的职业方向。
